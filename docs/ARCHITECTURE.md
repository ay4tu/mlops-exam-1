## 1. System Overview

ModelServe is a real-time fraud detection inference service built on top of a credit card transaction dataset. It exposes a REST API that accepts a credit card number (`entity_id`), looks up pre-computed transaction features from a Redis-backed Feast online store, runs a scikit-learn HistGradientBoostingClassifier loaded from MLflow, and returns a fraud probability score in under 10 ms.

The system separates concerns cleanly: training (session 01) produces a registered model in MLflow and a features Parquet file; the local serving stack (sessions 02–03) wires FastAPI, Feast, Redis, and MLflow into a Docker Compose topology for development and demo; production infrastructure (sessions 04–05) lifts the same containers to AWS EC2 behind an Application Load Balancer, with S3 for artifact storage and GitHub Actions for CI/CD.

The design philosophy is operational simplicity first: every component runs as a Docker container with a health check, startup ordering is explicit, and the serving path has no database calls — only a single Redis round-trip per request.

### ADR-3: Data Architecture

**Context:** MLflow needs a backend store for experiment metadata and a separate store for model artifacts. Feast needs an offline store for historical features and an online store for low-latency inference-time lookups.

**Decision:**
- MLflow backend store: PostgreSQL (via `psycopg2`)
- MLflow artifact store: Docker volume (`/mlruns`) in dev; S3 in production (Phase 4)
- Feast offline store: local Parquet file (`training/features.parquet`)
- Feast online store: Redis 7

**Rationale:**
- PostgreSQL is MLflow's recommended production backend; it supports concurrent writes from multiple training runs and persists across container restarts.
- Redis gives sub-millisecond key-value reads (O(1) per entity lookup). The serving path does exactly one Redis `GET` per prediction request — no SQL parsing, no index scans, no connection pooling overhead.
- Parquet as the offline store is zero-infrastructure for local development: the file is generated by `train.py` and read directly by Feast during materialization. There is no need to run a separate object store locally.
- S3 is deferred to production because it introduces an AWS dependency that isn't necessary during development or demo.

**Trade-offs:**
- Redis is in-memory only; features are lost on container restart. Mitigation: re-run `scripts/materialize_features.py` after each restart. In production, Redis persistence (AOF/RDB) would be configured.
- Local Parquet means the offline store can only be read from the machine where the file lives. In production this becomes S3 so any machine in the cluster can materialize.

### ADR-4: Containerization

**Context:** The FastAPI inference service needs to be packaged as a Docker image that is production-grade: minimal attack surface, non-root runtime, explicit health checks, and as small as feasible given the ML dependency stack.

**Decision:**
- Two-stage build: `python:3.11-slim` as builder (installs packages with gcc available), `python:3.11-slim` as runtime (only copies installed packages, no build tools)
- Non-root user `appuser` with no login shell
- `HEALTHCHECK` via `curl -f http://localhost:8000/health`
- Runtime server: gunicorn with 2 uvicorn workers
- Post-install surgical removal of `mypy`, `mypyc`, `bigtree` (feast bundles these for static analysis; they serve no runtime purpose)
- Serving-specific `requirements-app.txt` excludes training-only packages (`boto3`, `psycopg2-binary`, `pandas`, full `mlflow`) and uses `mlflow-skinny` instead

**Rationale:**
- `python:3.11-slim` over Alpine: Alpine uses musl libc which causes binary compatibility issues with pre-built wheels for scipy, pyarrow, and sklearn. slim gives a smaller image than full Python without the musl incompatibilities.
- Python 3.11 specifically: scikit-learn 1.8.0 (matching the training environment) requires Python ≥ 3.11.
- Multi-stage build prevents gcc, libpq-dev, and pip cache from appearing in the final image layer.
- `mlflow-skinny` replaces full `mlflow` to drop the MLflow UI server, Alembic migrations, and Databricks integration (~20 MB saved).

**Trade-offs — image size:**
The 800 MB target stated in the session plan is not achievable with `feast[redis]` + `scikit-learn` in the same image. scikit-learn imports `scipy.sparse` at module load time, making scipy (~136 MB) non-removable. feast imports dask at module level and uses pyarrow for feature serialization, adding another ~157 MB. These are mandatory runtime dependencies of the chosen ML stack, not packaging inefficiency.

Final image size: **1.21 GB** (down from 1.49 GB before optimization).
Savings breakdown: removed boto3 (~29 MB), psycopg2 (~15 MB), matplotlib/PIL/fontTools (~55 MB), pandas (~81 MB), mypy/bigtree/mypyc (~116 MB), full mlflow → skinny (~20 MB).
